## 1. 阶段结论

Phase 0.2 已正式标记为：

```text
COMPLETE
```

以下标签正式锁定为后续开发基线：

```text
baseline-writing-buddy-0.2-windows
```

Phase 0.3 的唯一目的，是验证 Writing Buddy 作为写作软件时的工作台布局与基础操作体验。

本阶段不是完整写作系统开发阶段。

## 2. 基线创建规则

开始 Phase 0.3 前必须完成以下检查：

```text
当前工作树干净
Phase 0.2 标签存在
Phase 0.2 标签可正确解引用
标签解引用目标为：
af858884cd982b88019c30fc76014c3ba0d921a5
```

目标分支必须使用以下方式从标签创建：

```powershell
git switch --detach baseline-writing-buddy-0.2-windows
git switch -c writer/phase-0.3-writer-workbench-layout
```

也可以使用语义等价的单条 Git 命令，但必须确保分支起点为标签解引用后的提交，而不是当前工作目录中的其他提交。

创建分支后必须再次验证：

```powershell
git rev-parse HEAD
git rev-parse baseline-writing-buddy-0.2-windows^{}
git status --short
```

预期结果：

```text
HEAD 与标签解引用目标一致
工作树为空
尚未修改任何产品源码
```

禁止移动、重建、覆盖或删除 Phase 0.2 标签。

## 4. 阶段目标

Phase 0.3 需要验证以下核心假设：

> 用户启动 Writing Buddy 后，第一眼看到的是作品、章节和正文，而不是文件系统、源代码和开发工具。

本阶段必须形成一个可以运行和操作的写作工作台原型，包括：

1. 左侧作品结构导航。
2. 中间 Monaco 正文编辑区。
3. 右侧静态写作辅助面板。
4. 顶部或底部写作状态信息。
5. 示例章节切换。
6. 当前章节与作品总字数统计。
7. 工作区和最后活动章节恢复。
8. 中文输入、保存和内容完整性验证。

## 5. 技术方案冻结

### 5.1 可选方案比较

#### 方案 A：深度修改 Code-OSS Workbench

直接修改大量 Workbench、Explorer、Activity Bar、Editor 和 Layout 源码。

优点：

```text
界面控制能力最强
最终产品形态自由度最高
```

缺点：

```text
改动范围大
容易影响 Code-OSS 原有稳定性
升级和维护成本高
不符合 Phase 0.3 原型验证定位
```

本阶段不采用。

#### 方案 B：完全使用内置扩展实现

只通过内置扩展提供 Tree View、Webview View 和状态栏。

优点：

```text
侵入性最低
容易隔离和回滚
```

缺点：

```text
难以可靠控制首次启动布局
容易仍然呈现为装有写作扩展的 VS Code
工作台产品感不足
```

本阶段不单独采用。

#### 方案 C：最小 Workbench 配置与内置写作扩展组合

由少量 Workbench 配置负责默认布局、可见区域和启动行为。

由一个内置写作原型扩展负责：

```text
作品结构导航
章节切换
静态辅助信息
字数统计
必要的状态记录
```

优点：

```text
能够形成明显的写作软件形态
保留 Monaco 和 Code-OSS 成熟能力
源码侵入范围可控
便于后续替换或扩展
```

缺点：

```text
需要明确 Workbench 与扩展之间的职责边界
需要同时验证布局恢复和扩展状态
```

### 5.2 最终选定方案

Phase 0.3 冻结采用：

```text
方案 C：最小 Workbench 配置 + 内置写作原型扩展
```

不得在实现过程中未经批准改为深度 Workbench 重构。

## 6. 工作台布局

目标布局为：

```text
┌──────────────────────────────────────────────────────────────────────┐
│ Writing Buddy / 当前作品 / 当前章节 / 保存状态 / 专注模式占位入口   │
├────────────────┬───────────────────────────────┬─────────────────────┤
│                │                               │                     │
│ 作品结构导航   │         Monaco 正文编辑区     │ 静态写作辅助面板    │
│                │                               │                     │
│ Volume 01      │ Chapter 001.md                │ Current Scene       │
│  Chapter 001   │                               │ Characters          │
│  Chapter 002   │ 正文内容                      │ Scene Goal          │
│  Chapter 003   │                               │ Notes               │
│ Characters     │                               │                     │
│ Worldbuilding  │                               │                     │
│ Timeline       │                               │                     │
│ Notes          │                               │                     │
├────────────────┴───────────────────────────────┴─────────────────────┤
│ 当前章节字数 / 作品总字数 / 行列位置 / 保存状态                     │
└──────────────────────────────────────────────────────────────────────┘
```

布局优先级为：

```text
正文编辑区 > 作品导航区 > 写作辅助区
```

窗口宽度不足时，必须优先保证正文编辑区可用。

右侧辅助区可以缩窄、折叠或隐藏，但不得导致正文编辑区严重受压。

## 11. 字数统计规则

为避免中文、英文和标点统计结果不一致，Phase 0.3 统一采用以下规则：

### 11.1 中文及东亚文字

每个有效的中文、日文或韩文文字字符计为一个字。

### 11.2 英文与数字

连续的英文字符或数字组合计为一个词。

示例：

```text
Writing Buddy 2026
```

统计为：

```text
3
```

### 11.3 不计入内容

以下内容不计入字数：

```text
空格
换行
制表符
纯标点符号
Markdown 标记字符
```

Phase 0.3 不要求对全部 Markdown 语法进行完整语义解析。

### 11.4 当前章节字数

当前章节字数必须基于 Monaco 当前内存中的文档内容实时计算。

尚未保存的修改也必须反映在当前章节字数中。

允许使用短暂防抖，但用户停止输入后应快速更新。

### 11.5 作品总字数

作品总字数为演示作品中三个章节的字数总和。

当活动章节存在未保存修改时，总字数必须使用活动编辑器中的最新内容覆盖其磁盘版本，避免当前章节字数与总字数明显不一致。

本阶段不统计 Characters、Worldbuilding、Timeline 和 Notes 中的内容。

## 12. 状态恢复规则

Phase 0.3 必须验证两次启动之间的状态恢复。

首次启动后，用户应能够：

```text
打开演示作品
切换至任意章节
编辑中文正文
保存或保留编辑状态
正常关闭程序
```

第二次启动时必须恢复：

```text
最后使用的工作区
最后活动的章节
写作工作台的基本区域
```

以下内容优先复用 Code-OSS 已有状态恢复机制：

```text
工作区恢复
已打开编辑器恢复
光标位置恢复
滚动位置恢复
编辑器未保存状态恢复
```

不得为了 Phase 0.3 创建完整的自定义项目状态数据库。

右侧面板宽度、左侧面板宽度等细节可以依赖现有 Workbench 布局存储，不要求开发新的布局序列化系统。

## 18. 中文编辑安全要求

必须创建或复用一个专门的中文验收章节。

验收前记录该文件 SHA-256。

执行以下操作：

```text
启动 Writing Buddy
打开中文章节
输入中文
删除中文
撤销
重做
复制
粘贴
保存
关闭
重新启动
再次打开章节
```

最终确认：

```text
文字内容符合预期
无乱码
无意外换行转换导致的内容损坏
文件可以正常重新读取
字数统计结果可以解释
```

对于要求内容保持不变的测试文件，验收前后 SHA-256 必须一致。

对于明确执行保存修改的文件，应保存修改前、修改后哈希及预期文本差异，不能错误地要求哈希一致。
