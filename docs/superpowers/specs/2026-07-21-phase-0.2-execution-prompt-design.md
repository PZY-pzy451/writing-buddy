# Phase 0.2 Downstream Repository and Minimal Branding Design

## Status

Approved on 2026-07-21. This document defines the scope and verification model for the single-run Codex execution prompt. It does not authorize Phase 0.3 work.

## Objective

Create a local downstream repository topology, establish an isolated Phase 0.2 branch, replace the minimum Windows desktop product identity with Writing Buddy, and re-run the Windows build and launch acceptance checks before creating a recoverable commit and annotated tag.

## Selected Approach

Use product-configuration-driven branding. Modify only the product identity fields needed for visible Writing Buddy naming and Windows desktop isolation, then regenerate the Electron runtime through the repository's official task.

Rejected alternatives:

- UI-only name overrides leave data directories, URL handling, mutexes, and Windows identity in conflict with Code-OSS.
- Full cross-platform branding expands the task into unverified macOS, Linux, server, tunnel, icon, packaging, and publishing work.

## Repository Topology

The existing Microsoft remote becomes `upstream`. A new local bare repository at `D:\develop_tool\writing-buddy-downstream.git` becomes `origin`.

```text
https://github.com/microsoft/vscode.git
                │
                └── upstream

D:\develop_tool\writing-buddy-downstream.git
                │
                └── origin
```

Rules:

- Never push to `upstream`.
- If the bare-repository path already exists, validate that it is a Git bare repository before using it; do not overwrite an unknown directory.
- Create the Phase 0.2 branch from the current verified repository history that includes the approved design document while preserving the Phase 0.1 tag as the immutable runtime baseline.
- Push only the Phase 0.2 branch and its final annotated tag to the local `origin`.

| `sharedDataFolderName` | `.writing-buddy-shared` |

| `win32DirName` | `Writing Buddy` |
| `win32NameVersion` | `Writing Buddy` |
| `win32RegValueName` | `WritingBuddy` |
| `win32AppUserModelId` | `WritingBuddy.Desktop` |
| `win32ShellNameShort` | `Writing &Buddy` |
| `urlProtocol` | `writing-buddy` |

Generate and persist four distinct GUIDs for:

- `win32x64AppId`;
- `win32arm64AppId`;
- `win32x64UserAppId`;
- `win32arm64UserAppId`.

The GUIDs must use the format expected by the existing product file and must not reuse Code-OSS identifiers. For each field, read the original value, extract the non-GUID prefix and suffix, and preserve that wrapper exactly; do not assume the number of braces or choose a new casing convention. Verification must prove that all four embedded GUIDs are legal, pairwise distinct, different from every original Code-OSS AppId, and wrapped exactly like their corresponding original fields.

## Explicitly Preserved Upstream State

Do not change:

- application icons or icon resources;
- MIT license names, files, URLs, or upstream attribution;
- Microsoft built-in extension metadata;
- Marketplace, update, telemetry, authentication, or extension-feed behavior;
- macOS, Linux, server, or tunnel identity fields;
- workbench layout, Activity Bar, editor behavior, or any Phase 0.3 feature;
- source files merely to replace incidental comments, tests, pipeline names, dependency package names, or historical `code-oss` strings.

The acceptance check must be field-based and runtime-based, not a global text replacement.

## Build and Runtime Design

Changing `nameShort` makes `scripts\code.bat` look for `.build\electron\Writing Buddy.exe`. The prelaunch task checks only whether the installed Electron version matches and therefore can incorrectly reuse the previously branded runtime.

The execution prompt must explicitly run the official `npm run electron` task after changing `product.json`. That task owns cleanup and regeneration of `.build\electron`; the prompt must not implement a separate broad deletion command.

Before regeneration, identify the Phase 0.1 test instance and watch process from their recorded executable paths, profile paths, process IDs, and logs. Close the test application through its UI, confirm that no unsaved test edit remains, and stop only the recorded baseline watch process if it would conflict with a fresh Phase 0.2 watch. Do not terminate unrelated Code-OSS, VS Code, Node, or Electron processes.

After Electron regeneration:

1. Start a fresh `npm run watch` process and capture Phase 0.2 output.
2. Wait for the complete main-source and built-in-extension compilation markers.
3. Require zero blocking TypeScript or extension errors and a live watch process.
4. Verify `.build\electron\Writing Buddy.exe` exists.
5. Launch `scripts\code.bat` with new isolated Writing Buddy user-data, shared-data, and extensions directories.
6. Verify the visible window title and executable identity use Writing Buddy.
7. Run `scripts\code-cli.bat --version` and require exit code 0.
8. Open the Phase 0.1 test workspace and persisted Chinese file.
9. Inspect Developer Tools for a continuously repeating fatal error.
10. Close only the test instance, launch a second time, and verify the workspace and file reopen.

## Data Isolation

Runtime validation must not reuse the Phase 0.1 Code-OSS profile directories. Freeze the Phase 0.2 runtime root as:

```text
D:\develop_tool\writing-buddy-runtime\
├── user-data\
├── shared-data\
├── extensions\
└── logs\
```

Use `user-data` and `extensions` through their supported command-line arguments. Before using `shared-data`, inspect the current Code-OSS source and CLI option definitions to verify how shared-data isolation is implemented. Use a supported argument only when the current source proves it exists; otherwise verify isolation through the configured `sharedDataFolderName` and the actual generated directory. Record the evidence in the build report rather than assuming an option.

The test may reuse `D:\develop_tool\writing-buddy-test-workspace` because it is content data rather than application profile state. Its existing `中文输入测试.txt` content must remain unchanged.

Record the test file's SHA-256 before any product modification and again after the second-launch verification. The two hashes must be identical.

## Failure Handling

- Preserve complete logs for every failed command.
- Diagnose the first root cause, apply the smallest in-scope repair, and rerun the failed step.
- Do not hide warnings, alter source to suppress diagnostics, or claim success after a failed exit code.
- Do not delete or overwrite existing non-generated repositories, profiles, logs, or user files.
- Generated Electron output may be replaced only through the repository's official regeneration task.
- If the local bare origin path is occupied by an unknown or non-bare directory, stop repository mutation, finish all safe diagnostics, and report the single required user decision.

## Verification Contract

Phase 0.2 passes only when all of the following are true:

- `upstream` points to `https://github.com/microsoft/vscode.git`.
- `origin` points to the validated local bare repository.
- The current branch is `writer/phase-0.2-minimal-branding`.
- Only the approved `product.json` identity fields and durable documentation changed.
- `product.json` parses as JSON and every required value matches the branding field contract.
- All four Windows AppIds are new, distinct GUIDs.
- Node still matches `.nvmrc`.
- Dependencies remain installed.
- A fresh watch build completes without blocking errors and remains alive.
- `.build\electron\Writing Buddy.exe` exists.
- The first Writing Buddy development window remains running and shows the new name.
- CLI version execution exits 0.
- Developer Tools has no persistent fatal error loop.
- The first test instance closes cleanly.
- The second launch succeeds with the isolated Writing Buddy profile.
- The existing Chinese test file reopens with its content unchanged.
- The Chinese test file SHA-256 after the second launch exactly matches the pre-change SHA-256.
- A Phase 0.2 build report records environment, commands, paths, issues, risks, and evidence.
- The final commit contains only the intended product configuration and documentation.
- The annotated Phase 0.2 tag targets the final commit.
- The branch and tag are present in the local bare `origin`.
- The final working tree is clean.
